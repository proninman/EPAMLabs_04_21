Тест кейс №1
Название тест кейса:  Нагрузочное тестирование сайта  банка
Предусловия тест кейса: Наличие специализированного ПО для эмуляции множества пользователей, наличие пар логин\пароль для идентификации пользователя в системе, наличие верифицированных счетов для переводов между ними.
Шаги тест кейса:
Шаг 1: Использовав специализированное ПО для эмуляции одновременной работы 2000 пользователей
Шаг2:Открыть главную страницу.
Шаг 3:Перейти на страницу личного кабинета.
Шаг4: Войти в систему.
Шаг5:Проверить состояние своих счетов.
Шаг6:Оплатить какую-либо услугу.
Шаг7:Выполнить перевод на какую-либо карту.
Шаг8:Выполнить выход из системы.
Ожидаемый результат тест кейса
Действия всех 2000 пользователей будут успешно обработаны. 

Тест кейс №2
Название тест кейса: Тестирование безопасности  мобильного приложения
Предусловия тест кейса: Наличие специализированного ПО для проверки безопасности, наличие пар логин\пароль для идентификации пользователя в системе
Шаги тест кейса:
Шаг 1:Проверка регистрации .
Шаг2:Проверка авторизации.
Шаг 3:Проверка восстановления пароля.

Шаг5:Проверка невозможности авторизации при неправильном логине.
Шаг6:Проверка трансакций.
Шаг7:Проверка продолжения сессии после ее разрыва по каким-либо причинам.

Тест кейс №3
Название тест кейса: Проверка ссылок веб-сайта на предмет целостности( существования страниц, на которые ведут данные ссылки)
Предусловия тест кейса: Наличие специализированного ПО для автоматического переходя по ссылкам, наличие пар логин\пароль для идентификации пользователя на сайте.
Шаги тест кейса:
Шаг 1:Проверка регистрации .
Шаг2: Переход по ссылкам в автоматическом режиме для нахождения "битых" ссылок.
Шаг3:Завершение работы и выход 
Ожидаемый результат тест кейса
При наличии "битых" ссылок они будут найдены и запротоколированы.

Тест кейс №4
Название тест кейса: Проверка возможности авторизации на большом количестве однотипных сайтов веб-сайта 
Предусловия тест кейса: Наличие специализированного ПО для автоматического переходя по ссылкам, наличие пар логин\пароль для идентификации пользователя на сайте.
Шаги тест кейса:
Шаг 1:Переход с помощью специализированного ПО на первый сайт в списке.
Шаг 2:Проверка возможности авторизоваться на этом сайте.
Шаг 3:Завершение сессии на этом сайте.
Шаг 4:Переход на следующий сайт списка.
Шаг 5:Повторении шагов 1-4 до исчерпания списка.


Шаг6:Формирование списка сайтов с информацией по возможности авторизации на них.
Шаг8:Завершение работы . 
Ожидаемый результат тест кейса
Создание списка сайтов с указанием на возможность\невозможность авторизации на них.

Тест кейс №5
Название тест кейса: Тестирование возможных конфигураций  консольного приложения.
Предусловия тест кейса: Наличие специализированного ПО для автоматического тестирования, наличие конфигурационного  файла с большим количеством параметров.
Шаги тест кейса:
Шаг 1: Определение последовательности изменения настроек в конфигурационном файле .
Шаг 2:Проверка первичной комбинации настроек .
Шаг 3:Запись в лог о состоянии системы при данном изменении.
Шаг 4:Изменение следующего значения параметра.
Шаг 5:Изменение параметра.
Шаг 5:Повторении шагов 1-5 до исчерпания списка параметров.
Ожидаемый результат тест кейса
Создание списка изменений параметров с описаниями влияния этих изменений на работоспособность системы.
